Why We Built This?

  • Bluetooth Inefficiency: Extremely slow speeds (< 1MB/s) for modern large files.
  • Internet Dependency: Cloud sharing (Drive/WhatsApp) fails in remote or restricted zones.
  • The App Barrier: Existing apps like Xender/ShareMe require both parties to have the app.
  • Bloatware: Most sharing apps are cluttered with intrusive ads and data tracking.

The FastShare Solution

Core Concept

FastShare transforms the sender's device into a Localized Web Server.

The receiver simply connects to the sender's network and uses a Web Browser to download files instantly.

Unique Value Proposition

  • No app installation needed for receiver.
  • Purely offline - No data consumption.
  • Universal compatibility (Android, iOS, PC).

Technical Architecture

The Client-Server Model

1. Sender Side: Acts as the Host. Spawns a HttpServer instance on a dynamic port (e.g., 54321).

2. Receiver Side: Acts as the Client. Connects via standard HTTP/1.1 protocol over TCP/IP sockets.

3. Discovery: Uses QR Code to bridge the IP-Port handshake between devices.

Engineering Challenge: RAM Optimization

Problem: The 2GB+ Crash

Standard apps try to load files into RAM before sending, causing crashes on large transfers.

Solution: Chunked Binary Streaming

FastShare reads files in 64KB chunks and pipes them directly to the network socket. RAM usage remains constant (< 100MB) even for 50GB files.

Smart Multi-File Handling

On-the-Fly Zipping

When a user selects multiple files (e.g., 100 photos):

  • The server creates a Virtual Zip Stream.
  • The file is zipped while it is being sent.
  • The receiver gets one FastShare_Bundle.zip instead of accepting 100 individual prompts.

Security & Authentication

Session Isolation

Every transfer session generates a unique 32-byte Secure Token. Even if someone knows the IP, they cannot access files without the token.

Industrial Protection

  • SHA-256: Passwords are hashed with salt.
  • Local Tunnel: Data never touches any external server or cloud.
